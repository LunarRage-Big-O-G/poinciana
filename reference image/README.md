# Reference images — concepts and designs to model

This folder collects **concepts, directions, and designs we like** and want our websites to **feel like** or **learn from**. These are inspiration and modeling targets—not necessarily final brand law, unless a file is explicitly called out that way elsewhere.

Use what you add here to **capture taste**: layouts that work, typography we admire, interaction patterns, color stories, and overall mood we want to channel when we build.

## How to use references when building

1. **Pick the intent** — Note *what* you are modeling from each file (e.g. “airy hero + strong headline,” “dense dashboard nav,” “soft card shadows”). We adapt ideas; we do not have to copy every detail unless we decide to.
2. **Before you build** — Keep the relevant reference visible while implementing so decisions stay aligned with the direction we liked.
3. **Translate, don’t only imitate** — Extract patterns: hierarchy, spacing rhythm, type scale, corner radii, motion feel, palette relationships. Implement them in our own components and tokens so the site stays coherent with *our* product.
4. **Compare often** — After meaningful UI work, step back: does the result evoke the same clarity, energy, or calm as the reference? Adjust until the *relationship* between elements matches the spirit of the concept.
5. **Name files clearly** — Descriptive names help everyone find the right mood, e.g. `inspo-minimal-landing.png`, `concept-bold-typography.png`, `model-ecommerce-cart-flow.png`.
6. **Keep history when direction shifts** — Add new files or dated folders when we pivot; older concepts stay useful as context for *why* we moved.

## What belongs here

- Design concepts or explorations we want to pursue or riff on.
- Screenshots or exports of sites, apps, or decks we **like** and want to **model** (layout, type, color, motion, tone).
- Rough mocks, mood boards, and “north star” visuals—even if not production-final.
- Notes in the filename or a sibling `.txt` if something needs context (e.g. “only the nav pattern”).

## What does not belong here

- Huge unrelated asset dumps (stock libraries, personal photos) with no UI modeling purpose.
- Anything with secrets, credentials, or sensitive personal data visible in screenshots.

## For AI-assisted development

When using an assistant to build UI, **point to the specific reference** in this folder and say what to model (e.g. “spacing and card style from X, but our colors”). That steers output toward our chosen concepts instead of generic templates.
